## Step 4 â€” Increment the build number

Each upload to App Store Connect requires a unique build number â€” even rejected or unreleased builds consume their number. The build number lives in `CURRENT_PROJECT_VERSION` in the Xcode project settings.

To bump it:

- In Xcode: select the **GenerativeRadio** target â†’ **General** tab â†’ **Build** field.
- Or edit directly in `mobile/ios/GenerativeRadio.xcodeproj/project.pbxproj` â€” search for `CURRENT_PROJECT_VERSION` and increment the value.

The marketing version (`MARKETING_VERSION` / `CFBundleShortVersionString`) only needs to change when you want to show a new version number to users. When you do change it, also update `version` in `mobile/app.json` to match â€” that is what `expo prebuild` stamps into the native project.

> **Gotcha â€” Xcode edits silently ignored:** `expo prebuild` generates `mobile/ios/GenerativeRadio/Info.plist` with *hardcoded* `CFBundleShortVersionString` and `CFBundleVersion` values, and literal Info.plist values override the Xcode build settings. If the General-tab fields don't show up in your archive, check that Info.plist contains `$(MARKETING_VERSION)` and `$(CURRENT_PROJECT_VERSION)` for those two keys (fixed on 2026-06-12). After re-running `expo prebuild`, the hardcoded values come back and you must re-apply the variable references â€” or just edit the Info.plist literals directly.

## Encryption export compliance

Apple asks about encryption on every first upload. Since the app uses HTTPS (but no non-exempt encryption), this is pre-answered in `mobile/app.json` under `expo.ios.infoPlist`:

```json
"ITSAppUsesNonExemptEncryption": false
```

This propagates into `Info.plist` automatically when Expo generates the native project. Do not edit `Info.plist` directly â€” `mobile/ios/` is gitignored and regenerated by `expo prebuild`.
